## Abstract

A Self-Authoring Improvement Proposal (SAIP) is a draft ZIP whose **authorship** — problem statement, root-cause analysis, proposed change, risk assessment, and implementation sketch — is generated by the protocol itself, while the **decision** remains with human token-holder governance. The protocol continuously ingests performance telemetry, aggregates the per-block anomaly flags emitted by Cognitive Consensus (ZIP-0420), and when metrics breach governed thresholds it triggers a frozen LLM to compose a structured proposal in standard ZIP format. Every machine-authored draft passes through a mandatory **adversarial review gauntlet** (a panel of distinct models attempting to find failure modes and attack vectors) and a simulation gate before it is eligible to reach a human vote. Outcomes — accepted, rejected, and the realized post-deployment metrics — are curated into the DSO Experience Ledger (ZIP-0410), so future SAIP generation is conditioned on which kinds of proposals historically worked. This ZIP being **Meta** type, it specifies the SAIP lifecycle, the data structures, the trigger and review machinery, and the constitutional limits on autonomous authorship; it does not itself change consensus rules.

## Motivation

The conventional improvement pipeline is `researcher → paper → ZIP → vote → implementation → deploy`, bottlenecked end-to-end by human attention and measured in months to years. Meanwhile, Cognitive Consensus (ZIP-0420) already produces a continuous, structured, on-chain signal about where the protocol is underperforming (anomaly/fee/MEV flags), and the DSO Experience Ledger (ZIP-0410) already stores curated outcomes. The missing piece is the layer that turns that signal into well-formed, reviewable proposals automatically.

SAIPs add exactly that layer and nothing more:

1. **Compress proposal latency** from months to (potentially) hours, while leaving the human vote — the locus of legitimacy — fully in place.
2. **Standardize quality.** Every SAIP is emitted in the ZIP-0001 frontmatter + section format, with a mandatory machine-checkable structure, so governance reviews a consistent artifact.
3. **Learn what works.** The accept/reject/outcome history becomes context for future generation; proposal quality improves via HLLM-style experience accumulation rather than weight updates.

The hard problems are (a) preventing the protocol from authoring proposals that are fluent but harmful, and (b) bounding what an autonomous author is even permitted to propose. Both are addressed normatively in the Specification.

## Specification

### 1. Constitutional scope (normative limits on autonomous authorship)

A SAIP author MUST operate inside a governance-pinned **authorship charter**. The charter partitions the protocol's parameter and rule space into three classes:

| Class | Examples | SAIP author may… |
|-------|----------|------------------|
| **Tunable** | fee curve params, mempool ordering weights, `τ_sem`, training duty cycle (ZIP-0422), committee size `k` | propose changes; eligible for *expedited* track |
| **Structural** | new tx types, state-schema changes, consensus rule edits | propose changes; *standard* track only; supermajority + timelock |
| **Forbidden** | minting/monetary policy, slashing of specific addresses, charter self-amendment, removal of the human vote | MUST NOT author; generation is hard-blocked |

The charter itself is amendable only by human governance (never by a SAIP). An emitted SAIP that touches a Forbidden field is rejected pre-review with a logged constitutional violation and counts against the author model's standing (§6).

### 2. Telemetry ingestion

A rolling, content-addressed `TelemetryWindow` is committed per epoch:

```
TelemetryWindow {
    epoch:            u64
    metrics:          map<MetricId, MetricSeries>   // see below
    cog_flags:        AggregatedFlags                // from ZIP-0420 attestations
    governance_state: { active_charter_hash, recent_votes }
    window_root:      bytes32                        // Merkle root, anchored on-chain
}
MetricSeries { id, samples:[u64], ema, variance, slope }
```

Tracked `MetricId`s (extensible by charter) include: `tps`, `finality_latency_ms`, `gas_utilization`, `validator_uptime`, `slashing_rate`, `economic_velocity`, `mev_extracted`, `mempool_depth`, `reorg_rate`, `cognitive_challenge_rate` (Tier-3 disputes from ZIP-0420), `inference_revenue` and `training_duty` (from ZIP-0422). `cog_flags` aggregates ZIP-0420 per-block flags across the epoch (e.g., a sustained `FEE_SPIKE` flag rate).

### 3. Trigger conditions

SAIP generation is **event-driven**, not free-running, to bound cost and noise. A trigger fires when any governed predicate over `TelemetryWindow` holds, e.g.:

```
trigger if  metrics[finality_latency_ms].ema  > THRESH_LAT
         or metrics[slashing_rate].slope      > THRESH_SLASH_TREND
         or cog_flags[ANOMALY].rate           > THRESH_ANOM
         or metrics[mev_extracted].ema        > THRESH_MEV
```

Thresholds are charter parameters. A debounce (min epochs between SAIPs targeting the same `MetricId`, default 16) prevents proposal spam. Each trigger records `{epoch, predicate_id, window_root}` as the SAIP's provenance.

### 4. Data structure: `SAIP`

```
SAIP {
    saip_id:        bytes32          // = H(window_root ‖ predicate_id ‖ author_model_hash)
    target_class:   enum {TUNABLE, STRUCTURAL}      // FORBIDDEN -> never emitted
    provenance:     { window_root, predicate_id, cog_attestation_refs:[bytes32] }
    author_model:   bytes32          // frozen authoring model hash (ZIP-0410)
    draft:          ZipDraft         // standard ZIP frontmatter + sections (§5)
    spec_diff:      Diff             // machine-applicable param/rule diff
    risk_report:    RiskReport       // §5
    sim_result:     SimResult        // §6.2, populated by the simulation gate
    review_panel:   [ReviewVerdict]  // §6.1, one per adversarial reviewer
    status:         enum {GENERATED, REVIEWING, SIM, READY, ON_BALLOT,
                          ACCEPTED, REJECTED, BLOCKED_CONSTITUTIONAL}
    sig:            bytes65          // authoring node signature
}
```

### 5. Generated content (the `ZipDraft`)

The author model MUST emit, grounded in `provenance` (every claim about chain state references a `cog_attestation_ref` or a `StateMerklePath`, reusing ZIP-0420's evidence discipline):

1. **Problem statement** — natural language + the breached predicate, with evidence references.
2. **Root-cause analysis** — a ReasoningTrace (ZIP-0420 §4 schema) whose EVIDENCE nodes Merkle-verify against committed state. Ungrounded root-cause claims fail structural validation, exactly as in ZIP-0420.
3. **Proposed change** — prose plus a machine-applicable `spec_diff` (a typed diff over charter-Tunable params, or a textual diff against the rule module for Structural).
4. **Risk assessment** (`RiskReport`) — enumerated failure modes, attack vectors, and an explicit Backwards-Compatibility statement.
5. **Implementation sketch** — pseudocode or a code diff.

The draft MUST be a syntactically valid ZIP (frontmatter per ZIP-0001, sections present) or it is rejected at the structural gate.

### 6. The review gauntlet (mandatory before any human vote)

A SAIP cannot reach `ON_BALLOT` until it clears all of:

#### 6.1 Adversarial multi-model review (Open Question 2)

A panel of `R` (default 3) **distinct** reviewer models — distinct by `model_hash` and ideally by lineage — each receives the SAIP with an adversarial instruction program: *find the failure mode, the attack vector, the way this proposal looks reasonable but harms the protocol.* Each emits a `ReviewVerdict{ model_hash, verdict ∈ {PASS, FLAG, BLOCK}, findings, sig }`. The author model is **excluded** from its own review panel. A single `BLOCK` from any panelist returns the SAIP to the author for revision (bounded to `MAX_REVISIONS`, default 2) or kills it. Reviewer findings are attached and surfaced verbatim to governance — humans see the strongest case against the proposal, not just the proposal.

#### 6.2 Simulation gate

The `spec_diff` is applied in a deterministic shadow/fork-simulation against recent on-chain history (replay of the last `W_sim` epochs, default 256) and an adversarial scenario suite. `SimResult` reports projected deltas on every tracked metric plus invariant checks. A SAIP that violates a hard invariant (e.g., a settlement-conservation invariant from ZIP-0422, or a Cognitive-Consensus liveness invariant from ZIP-0420 §8) fails the gate and cannot be balloted.

#### 6.3 Author standing

Each `author_model` carries an on-chain **standing score**: a function of (accepted SAIPs that improved their target metric) − (rejected SAIPs) − (constitutional violations) − (BLOCK verdicts received). Low standing throttles an author's allowed SAIP rate. Standing is informational to governance and never overrides the human vote.

### 7. Governance handoff and outcome curation

A `READY` SAIP is placed `ON_BALLOT` under the standard ZIP/DAO process — including the track requirements from §1 (expedited timelock for Tunable; supermajority + longer timelock for Structural). Humans decide. After resolution:

```
outcome = ExperienceRecord {
    saip_id, decision ∈ {ACCEPTED, REJECTED},
    pre_metrics:  snapshot,                 // at ballot time
    post_metrics: snapshot @ +N epochs,     // realized, only if ACCEPTED+deployed
    delta_vs_projection,                    // realized vs SimResult
    governance_rationale: optional CID
}
```

`outcome` is curated into the DSO Experience Ledger (ZIP-0410) and becomes retrieval context (`experience_refs`) for future SAIP generation. This is the HLLM loop applied to governance: the *voting model stays frozen* (humans), and *proposal quality improves through accumulated, content-addressed experience*, not through weight drift.

### 8. ZIP-number allocation

SAIPs that reach `ACCEPTED` are minted into the canonical ZIP series in the appropriate category range (per the ZIP-0001 process), with an `authored-by: SAIP` provenance field added to frontmatter so machine-authored ZIPs are distinguishable in the record. The AI range (400-499) is the natural home for AI-category SAIPs.

## Rationale

**Why Meta type, not Standards Track?** SAIPs change *how proposals are authored*, not the consensus state machine. The artifacts they produce are themselves Standards-Track/Meta ZIPs that go through normal review. Classifying this proposal as Meta keeps the authorship layer cleanly separated from the rules it may propose to change.

**Why keep the human vote?** Legitimacy. Autonomous authorship compresses the *expensive, slow* part of the pipeline (drafting, analysis, risk enumeration) while preserving the *legitimacy-bearing* part (the vote). "The LLM proposes; humans decide" is a hard constitutional invariant (§1 Forbidden class includes "removal of the human vote").

**Why the adversarial gauntlet?** A single fluent model will reliably produce proposals that *read* well; the dangerous failure is a plausible proposal that introduces a subtle vulnerability. Requiring distinct reviewer models to attack the draft, plus a simulation invariant gate, plus surfacing the reviewers' strongest objections to voters, turns the human vote into an informed decision rather than a rubber stamp on fluent prose. This directly answers Open Question 2 of the source draft.

**Why event-driven triggers and debounce?** Free-running generation would flood governance and burn compute. Tying generation to breached, governed predicates and debouncing per-metric keeps SAIP volume proportional to actual protocol stress.

**Why grounded evidence (Merkle paths / attestation refs)?** Reusing ZIP-0420's evidence discipline means a SAIP's root-cause analysis is verifiable the same way a CognitiveAttestation is: fabricated justification is detectable, cheaply, before review.

**Why charter classes?** Not all changes carry equal risk. Letting tunable-parameter SAIPs move on an expedited timelock (still human-voted) while forcing structural changes onto the slow, supermajority track matches procedural friction to blast radius, and the Forbidden class makes monetary/constitutional capture impossible to even draft autonomously.

## Backwards Compatibility

- SAIPs are **purely additive** to the ZIP-0001 governance process. A network can adopt SAIPs and continue accepting human-authored ZIPs unchanged; the only frontmatter addition is the optional `authored-by: SAIP` provenance tag, which legacy tooling can ignore.
- No consensus rule changes are introduced by this ZIP. Telemetry windows and SAIP objects live in an auxiliary namespace; nodes that ignore them still reach state consensus.
- **Bootstrap (Open Question 5):** before the Experience Ledger holds governance outcomes, SAIP generation runs in *advisory-only* mode (drafts produced and reviewed but not balloted) and/or is seeded from the Zen Agentic Dataset's blockchain-engineering corpus to provide initial generation context. The accept/reject loop activates once a minimum outcome history accrues.
- The charter (§1) ships disabled-by-default; a network must explicitly ratify a charter (human governance action) before any SAIP can be balloted.

## Security Considerations

1. **Adversarial proposals (the core threat, Open Question 2):** mitigated by §6.1 multi-model adversarial review (distinct models, author excluded, single BLOCK halts), §6.2 simulation/invariant gate, and §7 outcome accountability via author standing. The human vote, with reviewer objections surfaced, is the final backstop.
2. **Constitutional capture:** the Forbidden class (§1) hard-blocks monetary-policy, targeted-slashing, vote-removal, and self-amendment authorship; charter amendment is human-only. A SAIP attempting a Forbidden change is logged and penalizes author standing.
3. **Governance spam / attention-DoS:** event-driven triggers, per-metric debounce, author-standing throttling, and the requirement to clear the gauntlet before balloting bound the rate of human-facing proposals.
4. **Reviewer collusion / monoculture:** reviewers MUST be distinct by `model_hash` and SHOULD be distinct by lineage; the charter SHOULD require at least one reviewer outside the author's family. If all available models share lineage, the simulation gate (§6.2) and human vote remain independent defenses.
5. **Telemetry poisoning:** metrics derive from committed chain state and ZIP-0420 attestations (themselves verified/slashable); an attacker would have to corrupt consensus-level data to bias triggers, which is out of scope for SAIPs and inherited from base-layer security.
6. **Sim/real divergence:** `delta_vs_projection` (§7) is recorded; persistent over-optimism by an author erodes its standing and is visible to governance. The sim gate is a *necessary* not *sufficient* check — it can reject but never auto-accept.
7. **Self-reinforcing bias:** because generation is conditioned on past accepted SAIPs, the loop could entrench one style. Mitigations: outcome records include *rejected* and *under-performing* SAIPs (negative evidence), and the charter can mandate periodic human-authored proposals to keep the experience distribution from collapsing.
8. **Provenance forgery:** `saip_id` binds `window_root`, `predicate_id`, and `author_model_hash`; the authoring node signs the SAIP. A SAIP whose claimed provenance does not Merkle-verify against the committed `TelemetryWindow` is rejected.

## Reference Implementation (sketch)

```python
def saip_epoch_tick(window: TelemetryWindow, charter, ledger):
    for pred in charter.triggers:
        if pred.holds(window) and not debounced(pred, window.epoch):
            spawn_saip(window, pred, charter, ledger)

def spawn_saip(window, pred, charter, ledger):
    author = active_author_model()                       # frozen, ZIP-0410
    if pred.target_field in charter.FORBIDDEN:
        return log_violation(author, pred)               # constitutional hard-block
    ctx = ledger.retrieve(experience_refs_for(pred))     # past outcomes (HLLM context)
    draft = author.compose_zip(window, pred, ctx)        # grounded, evidence-referenced
    if not zip_structurally_valid(draft) or not evidence_grounded(draft, window):
        return discard(draft)
    saip = SAIP(provenance=prov(window, pred), author_model=author.hash,
                target_class=classify(draft.spec_diff, charter),
                draft=draft, spec_diff=draft.spec_diff,
                risk_report=draft.risk, status=REVIEWING)

    # 6.1 adversarial panel (author excluded, distinct models)
    panel = [m.adversarial_review(saip) for m in charter.reviewers if m.hash != author.hash]
    saip.review_panel = panel
    if any(v.verdict == BLOCK for v in panel):
        return revise_or_kill(saip, author)

    # 6.2 simulation / invariant gate
    saip.sim_result = shadow_fork_replay(saip.spec_diff, last_epochs=W_SIM)
    if violates_hard_invariant(saip.sim_result):         # incl. ZIP-0422 conservation
        return reject(saip, reason="invariant")

    saip.status = READY
    decision = governance.ballot(saip, track=track_for(saip.target_class, charter))
    record = ExperienceRecord(saip.saip_id, decision, snapshot_now(),
                              schedule_post_metrics(+N), saip.sim_result.projection)
    ledger.curate(record)                                # feeds future generation
    update_standing(author, decision, panel)
```

Components reused: ZIP-0420 ReasoningTrace schema + evidence-Merkle discipline + aggregated flags; ZIP-0410 Experience Ledger (retrieval + curation) and model registry; ZIP-0001 ballot/DAO process; ZIP-0422 settlement invariants for the sim gate.
